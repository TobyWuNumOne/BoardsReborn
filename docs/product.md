# Product Spec

## 產品定位

BoardsReborn 是真實店家流程數位化系統，第一版聚焦板類維修工單管理。系統以工單為操作中心，讓收件、報價、維修、完工、通知、取件可以被查詢與協作，並用條碼降低現場查詢與更新成本。

## 使用者

第一階段使用順序：

1. 自己先測試資料模型與流程。
2. 老闆、店員、維修人員實際使用。
3. 顧客透過公開查詢頁查看進度。

第一版只做單一管理者角色，不細分老闆、店員、維修人員權限。

## 痛點

- 進度依賴老闆記憶，其他人難以接手。
- 板子數量變多後，靠擺放位置已無法管理。
- 顧客會一直打電話詢問進度。
- 無法快速知道哪些板子快超時、哪些已完工、哪些放太久沒取。
- 現場仍以紙本工單為主，且已有流水號制度，需要低成本導入。

## MVP Scope

第一版管理以下物件：

- 顧客與聯絡方式。
- 工單上的板子快照資訊，例如板種、尺寸、顏色、貼紙或人工標記。
- 維修工單。
- 維修狀態與狀態歷史。
- 維修照片 metadata 與 Supabase Storage 路徑。
- 初始報價、追加項目、最終總價。
- 取件資訊、完工通知時間、待取件天數提示。
- 條碼查詢與批量狀態更新。
- 非同步標籤列印與補印任務。

第一版支援板子種類：

- 衝浪板。
- SUP。
- 雪板。

雪板不會有除濕流程。

第一版條碼能力適用於所有板類。條碼內容直接使用紙本工單號，不另建第二套主要識別碼。

第一版標籤列印不由平板瀏覽器直接控制 USB 標籤機。列印採非同步 print job，由固定桌機或樹莓派上的 Python Print Agent 控制 USB 熱感標籤機。

## 核心流程

1. 顧客送板。
2. 老闆或店員收件。
3. 若店員先收件，老闆之後補檢查與報價。
4. 填寫工單，沿用紙本工單號。
5. 系統建立以紙本工單號為 payload 的標籤列印任務。
6. 顧客確認。
7. 衝浪板或 SUP 若進水，可進入除濕中。
8. 開始維修。
9. 需要時記錄特殊工況與追加費用。
10. 修完並進入待取件。
11. 通知顧客。
12. 顧客取件。

狀態可跳階段，不強制線性流轉，但每次狀態變更都必須保留歷史。

## MVP 狀態

系統 enum 只保留第一版核心狀態：

- `RECEIVED`：已收件。
- `DRYING`：除濕中。
- `REPAIRING`：維修中。
- `READY_FOR_PICKUP`：待取件。
- `DELIVERED`：已交件。
- `CANCELLED`：已取消。

現場常用狀態可以先在 UI 中簡化顯示為：已收件、除濕中、維修中、已完工待取件、已交件、已取消。

「待取件超時」與「太久沒開工」不是正式狀態，而是列表標記與篩選條件：

- 待取件超時：通知完工後超過 14 天未取。
- 太久沒開工：已收件後超過店內設定天數仍未進入除濕或維修。

## 搜尋與列表

最常用搜尋：

- 工單號。

常見列表：

- 全部進行中工單。
- 待取件。
- 已完工未取。
- 超過預估完成日。
- 通知後超過 14 天未取。
- 已收件太久仍未開工。
- 今天新收件。
- 某位顧客的工單。

所有列表都必須支援 filter、sort、pagination。

## 顧客查詢

第一版顧客查詢使用：

- 紙本工單號。
- 顧客手機後四碼。

顧客可看到：

- 目前狀態。
- 預估完成日。
- 初始報價金額。
- 公開備註。
- 最近更新時間。

第一版顧客不可看影片。照片是否開放仍待確認；預設不顯示照片。

追加項目與原因需要被記錄，但預設作為內部與取件時的說明依據，不優先開放在顧客查詢頁。

## 使用情境與條碼工作流

- 平板為主：查詢工單、更新狀態、檢視列表、批量變更狀態。
- 手機為輔：新建工單、拍照、單張掃碼查詢。
- 單張掃碼：條碼內容就是 `paper_order_no`，掃到後可直接查詢該工單。
- 多張掃碼：前端可先累積多個 `paper_order_no`，再一次送出批量狀態更新。
- 導入方向以低摩擦、少輸入、掃碼加速為優先，不新增複雜批次管理概念。

## 現場硬體與列印

- 平板與固定桌機都使用 Nuxt Web App。
- 無線條碼槍以 keyboard wedge 模式輸入工單號，可設定掃描後送出 Enter。
- USB 熱感標籤機連接固定桌機或樹莓派。
- Python + systemd Print Agent 負責拉取待印任務、送出 TSPL/ZPL/EPL/DPL 指令、回報結果。
- 建立工單不要求同步列印成功；若列印失敗，可走補印流程。
- 補印是標準流程，不視為異常 workaround。

## 不做項目

第一版不做：

- 會計報表。
- 租借管理。
- 會員系統。
- 金流。
- 寄賣。
- 庫存管理。
- 多店管理。
- 顧客修改狀態。
- 影片上傳或影片顧客瀏覽。
- 平板瀏覽器直接控制 USB 標籤機。
- 綁死單一品牌標籤機 SDK。
- 一開始做完整微服務或複雜訊息佇列。
- 把標籤列印當一般文件列印流程。

一張板如果第二次送修，視為新的獨立工單事件。

## 成功指標

- 顧客打電話問進度次數下降。
- 老闆不用靠記憶管理板子。
- 任何人都能快速查到板子在哪個階段。
- 不同同事可以快速掌握進度並協作。
- 超過交期的工單能被標示出來。
- 不會忘記有板沒修。
- 收板到交件都有完整紀錄。

## 待與老闆確認

- 完工通知是否要做，以及使用電話、LINE、IG 或其他方式。
- 發生追加費用時，實際溝通方式與確認時點。
- 收件照、維修中照、特殊工況照、完工照是否有最低必拍規則。
- 顧客是否能看照片；若可以，哪些照片可看。
- 特殊工況照是否只限內部。
- 第一版寄板費是否只做提醒與備註。
- 完工後由誰負責通知顧客。
- 收錢時間與實際規則。
